# Shuiyuan Skill / 本地缓存分析系统技术规划

版本：`v0.1`  
状态：草案，待评审  
范围：仅技术规划，不涉及本次代码改动  

## 1. 文档目标

本文档用于明确 `shuiyuan_exporter` 后续的演进方向：

1. 解释当前项目的真实职责与运行逻辑。
2. 说明为什么“先抓取、后本地分析”的解耦方案更适合后续 skill。
3. 给出一套可增量更新、可缓存、可检索、可扩展的本地数据架构。
4. 明确 MVP、后续迭代路线、风险点和验收标准。

本文档的核心结论是：

- **当前项目适合做抓取与导出器，不适合直接承担长期分析系统的角色。**
- **后续应将系统拆分为：抓取层、规范化层、缓存层、检索层、分析层、输出层。**
- **Markdown 可以继续保留，但应降级为“导出格式”，不应作为主存储结构。**
- **图片在 MVP 中只需要保存、建立映射和按需提供给 LLM 读取，不强制做 OCR。**
- **视频 / 音频能力在第一阶段不是重点，可先延后。**

## 2. 当前项目现状

### 2.1 项目当前定位

当前仓库的主目标是：

- 读取饮水思源论坛帖子的内容；
- 将帖子导出为本地 Markdown；
- 下载图片；
- 将 Markdown 中的 `upload://...` 替换成更可用的本地路径或真实远端 URL。

当前主要文件职责如下：

- `main.py`：命令行入口、批量执行、单 topic 导出总流程。
- `utils.py`：Cookie 读取、Session 初始化、请求发送、分页并发、raw 文本修复与部分引用展开。
- `image_handler.py`：图片真实地址解析、图片下载、Markdown 图片路径替换。
- `attachments_handler.py`：附件真实 URL 替换。
- `audio_handler.py`：音频真实 URL 替换。
- `video_handler.py`：视频真实 URL 替换。
- `quality_list.py`：预定义 topic 列表，方便批量拉取。

### 2.2 当前主流程

当前单个 topic 的导出流程是：

1. 读取 topic 基本信息并获得标题；
2. 并发抓取 `raw/<topic>?page=N` 的所有 raw 文本页面；
3. 拼接为单个 Markdown 文件；
4. 再分页抓取 `t/<topic>.json?page=N` 的 cooked HTML；
5. 根据 cooked HTML 解析图片真实地址并下载图片；
6. 将附件 / 视频 / 音频的 `upload://...` 映射替换成真实 URL；
7. 最终输出为一个 Markdown 文件 + `images/` 目录。

也就是说，当前系统是一个典型的：

> **抓取 -> 修复 -> 替换 -> 导出**

流水线工具。

### 2.3 当前架构的优点

当前实现的优点很明显：

- 简单直接，容易跑通；
- 面向“备份单个帖子”这个目标足够有效；
- 已经具备 Cookie、并发分页、图片映射等论坛专用逻辑；
- 对 skill 来说，这些经验和代码非常有复用价值。

### 2.4 当前架构的局限

但如果目标变成“本地缓存 + 后续反复分析”，当前结构会出现明显局限：

1. **抓取和分析耦合**  
   当前抓到数据后直接导出 Markdown，缺少“原始数据持久缓存”和“规范化结构存储”。

2. **Markdown 不是理想主存储**  
   Markdown 更适合给人看，不适合增量更新、全文检索、结构化查询、版本管理。

3. **缓存仅是进程内短期缓存**  
   当前 `_request_posts_cache` 只在一次运行期间有效，程序退出后即消失。

4. **缺少 topic / post 级持久索引**  
   目前没有可供后续 skill 快速查询的本地数据库。

5. **媒体和文本没有统一数据模型**  
   图片下载了，但其与 post、raw、cooked 的对应关系没有沉淀为统一结构。

6. **无法优雅做增量更新**  
   对于持续更新的大楼，只能重跑或重新导出，代价较高。

## 3. 为什么要把“拉取”和“分析”解耦

### 3.1 真实使用场景不是“导出一次”

从后续 skill 的需求看，用户会不断提出这类问题：

- 这个 topic 能不能访问？
- 只看楼主最近一周说了什么？
- 楼里哪些帖提到了银行 / 电信 / 化工？
- 根据当前 topic，再找相关讨论帖；
- 帮我整理某个时间段的主题演化；
- 把一批帖子同步到本地，之后再慢慢分析。

这些问题都不适合每次重新联网抓取整个 topic。

### 3.2 大楼帖子需要“数据底座”

以 `351551` 为例：

- 帖子数量超过六千；
- 时间跨度接近一年；
- 图片数量很多；
- 后续还会继续更新；
- 你未来还希望基于它找“相关帖子”。

这类数据明显更像一个“本地知识库”场景，而不是“一次性导出文件”场景。

### 3.3 解耦后的收益

把抓取和分析分离后，会带来明显收益：

- **减少重复联网请求**：同一个 topic 不必每次都重抓；
- **支持增量更新**：只同步新增楼层；
- **支持多视角分析**：同一批缓存数据可被不同分析逻辑复用；
- **便于构建 skill**：skill 可以优先读本地缓存，只在必要时联网；
- **方便维护数据质量**：抓取问题、解析问题、分析问题可以分层定位。

## 4. 目标系统的设计原则

后续系统建议遵循以下原则：

### 4.1 原始数据与派生数据分离

必须保留网络返回的原始数据（raw page / json page / 单帖 raw），不要只保留最终 Markdown。

### 4.2 Markdown 只做导出，不做主存储

Markdown 可以继续生成，但它应该只是：

- 人类阅读视图；
- 导出副产物；
- 报告输出；

而不是系统内部的事实来源。

### 4.3 图片优先于音视频

结合当前需求优先级：

- 第一优先级：文字、图片、引用关系、时间线；
- 第二优先级：附件；
- 第三优先级：音频 / 视频。

MVP 中可以先把音视频能力弱化或延后。

### 4.4 图片先保存，再按需交给 LLM

对于图片：

- 先保证能解析真实地址、下载、索引、定位到所属 post；
- 后续如果某次分析真的需要理解图片，再把图片传给 LLM；
- **不要求在第一阶段对所有图片统一 OCR**。

### 4.5 本地存储必须支持增量更新

新的主结构必须天然支持：

- topic 级更新；
- page 级更新；
- post 级追加；
- 媒体去重；
- 历史数据保留；
- 可重建索引。

### 4.6 Skill 以“查询 / 总结 / 扩展”而不是“导出”作为核心目标

导出只是其中一种输出形式。Skill 的核心应是：

- 查询；
- 过滤；
- 总结；
- 找相关帖子；
- 进行结构化分析。

## 5. 推荐的总体架构

建议未来系统分为 6 层：

1. **抓取层（Fetcher）**
2. **规范化层（Normalizer）**
3. **缓存与索引层（Cache / Store）**
4. **媒体层（Media Resolver）**
5. **分析层（Analysis）**
6. **输出层（Renderer / Skill Adapter）**

整体流程如下：

```text
论坛 HTTP 接口
    -> 抓取层
    -> 原始响应落盘
    -> 规范化层
    -> SQLite / 文件缓存 / 全文索引
    -> 分析层
    -> Skill / Markdown / JSON / 报告输出
```

### 5.1 抓取层

负责：

- 读取 Cookie；
- 发请求；
- 控制重试、并发、限速；
- 取得 `topic.json`、`topic page json`、`raw page`、单帖 raw；
- 将原始结果落盘；
- 提供增量同步所需的基本状态。

### 5.2 规范化层

负责把不同接口返回统一整理成一个稳定数据模型，例如：

- topic 元信息；
- post 列表；
- 楼层作者与时间；
- raw markdown；
- cooked HTML；
- plain text；
- 图片 / 附件 / 引用引用信息。

### 5.3 缓存与索引层

负责：

- 存原始响应；
- 存规范化后的 topic / post / media 数据；
- 建全文索引；
- 记录同步状态；
- 提供快速查询。

### 5.4 媒体层

负责：

- 识别图片、附件、视频、音频；
- 生成统一 manifest；
- 处理图片下载与去重；
- 保存本地路径与远端 URL 映射。

### 5.5 分析层

负责：

- 只看楼主；
- 按作者 / 日期 / 关键词检索；
- 时间线总结；
- 高频主题分析；
- 相关帖子发现；
- 提供适合 skill 的高层接口。

### 5.6 输出层

负责：

- JSON 输出；
- Markdown 输出；
- 控制台摘要；
- 最终 skill 回复所需的格式。

## 6. 为什么不建议继续以 Markdown 作为主结构

### 6.1 Markdown 的优点

- 人能直接读；
- 已经有现成导出逻辑；
- 适合作为备份产物；
- 对单 topic 手动浏览很方便。

### 6.2 Markdown 的缺点

当它作为主存储时，问题会很明显：

1. 很难做 post 级增量更新；
2. 很难做作者过滤、日期过滤、关键词检索；
3. 很难保留 raw / cooked / plain text 三种视图；
4. 很难建立图片和楼层的强关联；
5. 很难做“只更新最后几页”的同步；
6. 很难作为相关帖子推荐与本地搜索的基础。

### 6.3 推荐做法

推荐把 Markdown 视为：

- `rendered_view`
- `export_artifact`
- `report_output`

而把主存储改为：

- 文件系统中的原始 page 数据；
- SQLite 中的结构化 records；
- 媒体 manifest + blob 文件。

## 7. 推荐的缓存结构

我建议第一阶段使用：

- **文件系统** 保存原始响应与媒体文件；
- **SQLite** 保存结构化索引与检索数据；
- **FTS5** 提供全文检索。

这是复杂度和实用性的最佳平衡点。

### 7.1 推荐目录结构

```text
cache/
  raw/
    topics/
      351551/
        topic.json
        sync_state.json
        pages/
          json/
            0001.json
            0002.json
          raw/
            0001.md
            0002.md
        posts/
          000001.raw.md
          000002.raw.md
  media/
    images/
      53/
        A8eQqRwvsuhBAfa7glJj14XyI.jpeg
    attachments/
    audio/
    video/
  db/
    shuiyuan.sqlite
  exports/
    markdown/
      351551/
        351551 牢A投资记录.md
        images/
```

说明：

- `raw/topics/<topic_id>/pages/json/` 存分页 JSON 的原始响应；
- `raw/topics/<topic_id>/pages/raw/` 存分页 raw 文本；
- `raw/topics/<topic_id>/posts/` 存必要时拉取的单帖 raw；
- `media/images/` 存图片实际二进制；
- `db/shuiyuan.sqlite` 存索引与查询数据；
- `exports/markdown/` 存可选导出结果。

### 7.2 为什么这结构支持增量更新

因为未来更新 topic 时，可以只更新以下部分：

- `topic.json`
- 新增的 `json page`
- 新增的 `raw page`
- 新增或变更的 `post` 数据
- 新增图片文件
- SQLite 中新增的 post 记录与索引

也就是说，原始页面和派生结果不再绑定成一个“大文件”，而是页面级、记录级、资源级的可更新结构。

## 9. 推荐的数据库方案

### 9.1 为什么选 SQLite

推荐第一阶段选择 SQLite，原因如下：

- 本地单机使用简单；
- skill 易于集成；
- 便于事务与索引；
- 足以承载 topic / post / media 关系；
- 可以直接使用 FTS5 做全文检索；
- 不需要额外部署服务。

### 9.2 建议的逻辑表

建议逻辑上至少有这些表：

- `topics`
- `posts`
- `media`
- `post_quotes`
- `sync_state`
- `topic_tags`
- `posts_fts`

### 9.3 全文检索建议

建议用 SQLite FTS5 对以下文本建立全文索引：

- `raw_markdown`
- `plain_text`
- 视情况抽取后的引用文本

这样就能支持：

- 按关键词搜某个 topic；
- 按作者 + 关键词搜；
- 按时间 + 关键词搜；
- 在本地快速回答 skill 问题。

## 10. 图片与媒体策略

### 10.1 图片是第一优先级

第一阶段重点应该放在：

- 图片真实 URL 解析；
- 图片下载；
- 图片与 post 的映射；
- 图片本地路径持久化。

### 10.2 不强制对所有图片做 OCR

我建议默认策略是：

- **保存图片本体**；
- **记录图片属于哪个 post**；
- **后续分析时，按需将图片交给 LLM**；
- **只有在确有需要时，才补充 OCR 管线**。

这样做的好处是：

- 降低首版复杂度；
- 避免大量无价值 OCR；
- 保留更灵活的未来处理空间。

### 10.3 附件可保留映射，下载可延后

附件第一阶段可以：

- 先保存真实 URL；
- 再决定是否离线下载。

### 10.4 音视频降级处理

鉴于当前优先级，MVP 中可采用：

- 先记录元信息与 URL；
- 不作为主线功能投入大量精力；
- 等文字 + 图片主链路稳定后再补齐。

## 11. 增量更新方案

这是本规划的关键部分。

### 11.1 目标

对于已经抓过的 topic，下次同步时应尽量只做：

- 检查是否有新帖；
- 仅拉新增页；
- 仅更新变更区间；
- 仅下载新增图片；
- 仅更新数据库中受影响记录。

### 11.2 基本增量依据

建议主要依赖：

- `posts_count`
- `last_posted_at`
- topic 当前分页数

如果这几个值都没变，通常可以跳过 topic 级同步。

### 11.3 增量同步策略

#### 策略 A：Topic 级检查

先读一次 `topic.json`：

- 若 `posts_count` 未变化且 `last_posted_at` 未变化，则认为 topic 无需更新；
- 若变了，则进入增量同步。

#### 策略 B：Page 级追加

根据 `posts_count` 计算当前需要的 raw / json 总页数：

- raw 分页按 `100`；
- json 分页按 `20`；
- 只同步上次之后新增的页；
- 若最后一页可能变化，则回补最后 1~2 页以防漏数据。

#### 策略 C：Post 级去重写入

写入 SQLite 时，以 `post_id` 或 `(topic_id, post_number)` 做 upsert：

- 新帖插入；
- 已存在则比较更新时间 / 内容 hash 决定是否更新。

#### 策略 D：图片增量下载

对于图片：

- 若 `resolved_url + key` 已存在且本地文件存在，则跳过下载；
- 若不存在，则新增下载任务。

### 11.4 为什么这个方案适合 Shuiyuan

因为论坛帖子的增长天然是：

- 整个 topic 追加新楼层；
- 大多数旧楼层不变；
- 媒体在 post 内稳定绑定。

这非常适合 page 级 + post 级联合增量同步。

## 12. 推荐的分析能力

在有了本地缓存之后，skill 的分析层应优先实现以下能力。

### 12.1 楼内检索

支持：

- `topic_id`
- `keyword`
- `author`
- `date_from`
- `date_to`
- `post_number range`
- `only_op`
- `has_images`

### 12.2 主题摘要

支持：

- 整帖摘要；
- 指定时间段摘要；
- 只看楼主摘要；
- 高频关键词与话题演化；
- 最近 N 天更新摘要。

### 12.3 相关帖子发现

支持两种来源：

1. **论坛在线搜索**：通过 `search/query.json`、tag 页、latest 页；
2. **本地缓存搜索**：在已缓存的 topic 集合中找相关帖子。

### 12.4 作者视图

支持：

- 统计某个作者在某个 topic 中的发言；
- 汇总楼主观点变化；
- 抽取关键发言。

### 12.5 媒体关联视图

支持：

- 找出带图帖子；
- 找出截图最密集的时间段；
- 把图片一并提供给后续模型分析。

## 13. 未来 skill 的职责定义

未来的 Shuiyuan skill 建议不是一个简单的“导出器包装器”，而是一个：

> **本地优先、可增量同步、可检索、可总结、可扩展到相关帖子发现的分析型 skill**

### 13.1 MVP 阶段 skill 目标

第一版建议做到：

- 读取一个 topic URL / topic id；
- 检查本地缓存是否存在；
- 无缓存则抓取并建立本地记录；
- 有缓存则直接本地检索；
- 支持 `only_op`、关键词筛选、日期筛选；
- 支持输出摘要；
- 支持找相关帖子。

### 13.2 非目标（第一阶段暂不做）

第一阶段可明确不做：

- 全站镜像级别的海量同步；
- 自动登录流程（只用 Cookie）；
- 全量 OCR；
- 视频 / 音频的深度理解；
- 重型向量数据库与复杂分布式架构。

## 14. 与现有代码的关系

## 16. 风险与注意事项

### 16.1 Cookie 过期

这是最现实的问题。系统要明确区分：

- 网络错误；
- Cookie 过期；
- 页面权限不足；
- 请求过频。

### 16.2 请求频率

大楼同步会产生很多请求，尤其是：

- json 分页；
- raw 分页；
- 图片下载。

后续必须在抓取层加：

- 并发控制；
- 节流；
- 失败重试；
- 断点恢复。

### 16.3 数据一致性

同一个 topic 的 raw、json、media 可能分多次同步完成，必须允许：

- 部分同步成功；
- 媒体滞后于文本；
- 索引延后刷新；
- 最终一致性，而不是一次性全成。

### 16.4 历史可追溯性

若以后解析逻辑变化，最好能基于原始 page 文件重新构建规范化数据，因此原始响应不要丢。

## 17. 验收标准（建议）

以下标准可以作为后续实施时的验收依据。

### 17.1 缓存层验收

- 能对指定 topic 完成一次全量同步；
- 原始 json / raw 页面正确落盘；
- SQLite 中能看到 topic 和 post 记录；
- 重复同步不会无限重复写入。

### 17.2 增量更新验收

- 对已同步 topic 再次同步时，若无更新，能快速跳过；
- 若有新增楼层，只拉增量页；
- SQLite 中只新增或更新必要记录。

### 17.3 图片验收

- 能定位图片所属 post；
- 能下载图片；
- 能记录真实 URL 与本地路径；
- 不要求第一阶段 OCR。

### 17.4 查询验收

- 能按关键词检索某个 topic；
- 能只看楼主；
- 能按作者 / 时间过滤；
- 能输出结构化结果。

### 17.5 Skill 验收

- 能用 topic URL / id 作为输入；
- 能在本地缓存命中时直接回答；
- 能在无缓存时先同步再回答；
- 能给出相关帖子建议。

## 18. 当前建议的最终方案

综合当前需求和复杂度，建议采用如下方案：

### 18.1 主方案

- **主存储：文件缓存 + SQLite**
- **主更新方式：topic 级检查 + page 级增量 + post 级 upsert**
- **主媒体策略：图片优先下载并索引，OCR 延后**
- **Markdown 角色：导出与展示，不做主数据结构**
- **Skill 角色：本地优先分析，必要时增量联网**

### 18.2 第一阶段范围

第一阶段建议只聚焦：

- 文字；
- 图片；
- 本地缓存；
- 增量更新；
- 本地检索；
- 相关帖子发现。

### 18.3 暂缓项

第一阶段可以主动暂缓：

- 视频 / 音频深度处理；
- OCR；
- embedding / 向量检索；
- 自动登录；
- 超大规模全站索引。
